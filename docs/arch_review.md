## 1. Резюме

Проект — один host-скрипт, который поднимает одноразовую headless Ubuntu-ВМ в
VirtualBox с Node.js, Claude Code и Codex CLI через cloud-init. Для своего
размера скрипт написан качественно: `set -euo pipefail`, аккуратные проверки
зависимостей, продуманная экономия диска, честный README.

Главные системные слабости:

1. **Скрипт не идемпотентен и не убирает за собой при сбое** — упавший на
   середине запуск оставляет полусозданную ВМ/зарегистрированный диск, и
   повторный запуск падает уже по другой причине.
2. **«Успех» объявляется до фактической готовности** — скрипт завершается на
   `startvm`, не дожидаясь cloud-init; ошибки установки Node/CLI внутри гостя
   молча теряются, а MOTD «dev tooling ready» пишется даже при провале.
3. **Слабая цепочка доверия к артефактам** — образ Ubuntu не проверяется по
   контрольной сумме, NodeSource ставится через `curl | bash`, включён
   парольный SSH с дефолтным паролем `dev`.
4. **Проект «одноразовый» по духу, но пересоздание дорогое** — нет снапшота
   «pristine» после провижининга, каждый reset — это полный цикл заново.

Ни одна из проблем не блокирует текущее использование (ВМ локальная, порт
привязан к 127.0.0.1), но пункты 1–2 будут регулярно кусаться при повседневной
работе, а пункт 4 — главная точка роста ценности проекта.

## 2. Что сделано хорошо

- **Правильный выбор базы**: cloud image + NoCloud seed вместо ISO-инсталляции —
  минимальный диск, быстрый первый запуск, декларативный provisioning.
- **Тонкий VDI + автоувеличение rootfs** — экономия диска реально работает и
  честно описана в README.
- **Fallback-цепочка сборщиков seed ISO** (`cloud-localds` → `genisoimage` →
  `mkisofs` → `xorriso`) — снижает трение при установке на разных хостах.
- **Фикс `56cab3a` (npm prefix в `~/.npm-global`)** — правильное решение:
  CLI обновляются без root, нет столкновений с системным npm.
- **NAT port-forward привязан к `127.0.0.1`** (`provision-dev-vm.sh:154`) —
  SSH не торчит в локальную сеть.
- **Кэширование базового образа по имени релиза** (`provision-dev-vm.sh:57,60`) —
  повторные создания ВМ не качают образ заново.
- README краток, точен и синхронизирован со скриптом (на текущий момент).

## 3. Находки

Severity: 🔴 высокая (сломает или подставит), 🟡 средняя (регулярное трение),
🔵 низкая (гигиена).

### 3.1 Надёжность и идемпотентность

**🔴 R1. Нет отката при сбое — повторный запуск падает загадочно.**
`set -e` прерывает скрипт, но trap чистит только `$TMP`
(`provision-dev-vm.sh:70`). Если упал любой шаг после `createvm`
(`provision-dev-vm.sh:150`), остаётся зарегистрированная ВМ без дисков, и
повторный запуск умирает на проверке «VM already exists»
(`provision-dev-vm.sh:40`). Хуже: `qemu-img convert` перезаписывает
`$DISK` (`provision-dev-vm.sh:65`) с **новым UUID**, а старый UUID по тому же
пути может оставаться в media registry VirtualBox — `storageattach` тогда
падает с невнятной ошибкой про UUID.
*Рекомендация:* trap на откат (`unregistervm --delete` созданной ВМ при сбое) +
`VBoxManage closemedium disk "$DISK" 2>/dev/null || true` перед конвертацией.
Либо флаг `--force`, который сначала сносит одноимённую ВМ и её артефакты.

**🔴 R2. Успех объявляется до готовности; ошибки cloud-init теряются.**
Скрипт заканчивается на `startvm` (`provision-dev-vm.sh:162`) и печатает
инструкции. Если внутри гостя упал NodeSource, apt или `npm install`
(`provision-dev-vm.sh:129-131`), пользователь узнаёт об этом только когда
`claude: command not found`. Причём runcmd cloud-init выполняется как обычный
sh-скрипт **без** `set -e`: упал шаг 1 — остальные шаги (теперь их 5, включая
`update-grub` из `e912e2d`) всё равно выполняются, и
MOTD «dev tooling ready» (`provision-dev-vm.sh:133`) пишется в любом случае,
т.е. врёт.
*Рекомендация:* после `startvm` поллить SSH и выполнить
`cloud-init status --wait` в госте; проверить `command -v claude codex`;
завершаться ненулевым кодом при провале. MOTD писать только последним шагом
единого скрипта с `set -e` (или через `bootcmd`/один `runcmd`-элемент
`bash -ec '…'`).

**🟡 R3. Кэш образа: битые и протухшие файлы.**
Скачивание идёт прямо в `$BASE` (`provision-dev-vm.sh:62`); прерванный curl
оставляет частичный файл, который при следующем запуске считается валидным
(`[ ! -f "$BASE" ]`) — `qemu-img convert` упадёт или, хуже, соберёт битую ВМ.
Плюс URL указывает на `current/` — движущуюся цель, но кэш живёт вечно:
через полгода «свежая» ВМ поднимается на полугодовалом образе с сотнями
отложенных обновлений (а `package_upgrade: false`).
*Рекомендация:* качать в `$BASE.part` + `mv` по успеху; сверять SHA256 с
`SHA256SUMS` рядом с образом (решает и целостность, и supply chain — см. S3);
опционально — предупреждать/переcкачивать, если кэш старше N дней.

**🟡 R4. Нет проверки, что `SSH_PORT` свободен.**
Вторая ВМ с дефолтным портом 2222 создастся успешно, но SSH-форвард молча не
заработает (или отвалится при старте). Для сценария «несколько песочниц» это
первый же грабель.
*Рекомендация:* проверять порт (`ss -ltn`/`nc -z`) и либо падать с понятным
сообщением, либо автоподбирать свободный порт и печатать его.

**🟡 R5. `--resize` не умеет уменьшать.**
При `DISK_GB` меньше виртуального размера cloud-образа (~3.5 GB)
`modifymedium --resize` (`provision-dev-vm.sh:66`) упадёт. README при этом
советует «Drop DISK_GB to 12» — ок, но границу стоит валидировать явно с
понятной ошибкой.

**🔵 R6. Seed ISO остаётся примонтированным навсегда.**
После первой загрузки `seed.iso` больше не нужен, но остаётся в приводе
(`provision-dev-vm.sh:158`). Не вредит, но мешает удалить/пересобрать seed и
оставляет `instance-id` активным.
*Рекомендация:* в финальной инструкции (или в шаге health-check из R2)
отсоединять: `storageattach … --medium emptydrive`.

**🔵 R7. Не проверяются host-зависимости `curl` и `ssh-keygen`.**
Проверяются только `VBoxManage` и `qemu-img` (`provision-dev-vm.sh:38-39`),
хотя curl нужен на строке 62, а ssh-keygen — на 49.

### 3.2 Безопасность

**🔴 S1. Парольный SSH включён с дефолтным паролем `dev` + NOPASSWD sudo.**
`ssh_pwauth: true` (`provision-dev-vm.sh:86`) включает вход по паролю именно
по SSH, а README называет пароль «console password» — это неточно. Сейчас
риск смягчён привязкой к 127.0.0.1, но конфигурация «пароль dev + sudo без
пароля» становится дырой при первом же изменении сети (bridged, проброс на
0.0.0.0, доступ других пользователей хоста к localhost).
*Рекомендация:* `ssh_pwauth: false` — ключ и так авторизуется; пароль оставить
только для VirtualBox-консоли (это и есть заявленный сценарий). Идеально —
генерировать случайный `VM_PASS` и печатать его.

**🟡 S2. `curl | bash` от root для NodeSource (`provision-dev-vm.sh:129`).**
Классический supply-chain-риск: исполняем неверифицированный скрипт из сети с
root-правами. Прагматично, но есть более дешёвые по риску варианты.
*Рекомендация (по возрастанию усилий):* (а) хотя бы зафиксировать в README как
осознанный компромисс; (б) вручную добавить подписанный apt-репозиторий
NodeSource (keyring + sources.list — ~5 строк в cloud-config через `apt:`);
(в) ставить Node из официального tarball с проверкой SHASUMS.

**🟡 S3. Образ Ubuntu скачивается без проверки целостности/подлинности.**
См. R3 — рядом с образом публикуются `SHA256SUMS` и GPG-подпись. Проверка
суммы — одна строка, закрывает и битые загрузки, и подмену.

**🔵 S4. Секрет-гигиена мелочей:** пароль печатается в финальном выводе
(`provision-dev-vm.sh:169`); содержимое `PUBKEY_CONTENT`, `VM_USER`, `VM_PASS`
интерполируются в YAML без экранирования — пароль с `:`, `#` или кавычками
молча сломает cloud-config. Достаточно обернуть значения в двойные кавычки в
YAML и/или валидировать входные переменные.

**🔵 S5. Изоляция сети гостя — только NAT наружу.**
Для «песочницы под автономных агентов» полный исходящий интернет — осознанный
компромисс, но он нигде не проговорён. Стоит зафиксировать в README модель
угроз: от чего ВМ защищает (хост-ФС, хост-процессы), а от чего нет (эксфильтрация
в сеть, злоупотребление учётками, в которые агент залогинен).

### 3.3 Переносимость и совместимость

**🟡 P1. macOS-инструкции подразумевают Intel, а образ жёстко amd64.**
`IMG_URL` фиксирует `amd64` (`provision-dev-vm.sh:32`); на Apple Silicon
VirtualBox (7.1+, ARM) этот образ не загрузится. README рекламирует macOS без
оговорок.
*Рекомендация:* определить арх (`uname -m`), на arm64 — либо подставлять
`arm64`-образ (для VirtualBox ARM), либо честно умирать с сообщением. Минимум —
оговорка в README «macOS = Intel».

**🟡 P2. Флаги VBoxManage зависят от мажорной версии.**
`--audio-driver` появился в VirtualBox 7.x (в 6.x — `--audio`);
`--graphicscontroller`, `--firmware efi` тоже менялись. На хосте с VirtualBox
6.1 скрипт упадёт на `modifyvm` (`provision-dev-vm.sh:151-152`) с криптичной
ошибкой.
*Рекомендация:* проверять версию (`VBoxManage --version`) и требовать ≥7, с
внятным сообщением.

**🔵 P3. `chpasswd.list` — устаревшая форма схемы cloud-init.**
Начиная с cloud-init 22.x рекомендуется `chpasswd.users`; старая форма пишет
deprecation-warning и однажды отвалится при обновлении образа.

### 3.4 UX и сопровождение

**🟡 U1. Тихая генерация SSH-ключа мутирует `~/.ssh` хоста.**
Если у пользователя ключ под нестандартным именем (id_ecdsa, ключи в агенте),
скрипт молча создаст новый `id_ed25519` (`provision-dev-vm.sh:47-51`). Это
неожиданный побочный эффект вне заявленной зоны действия скрипта.
*Рекомендация:* спрашивать подтверждение либо генерировать ключ per-VM в
`$WORKDIR` и печатать готовую ssh-команду с `-i`.

**🟡 U2. Пересоздание ВМ ломает known_hosts.**
Каждая новая ВМ на том же `127.0.0.1:2222` даёт новый host key → страшное
предупреждение MITM и ручная чистка known_hosts. Для «disposable»-воркфлоу это
постоянное трение.
*Рекомендация:* печатать (или генерировать `~/.ssh/config.d`-сниппет) блок:
`Host dev-vm → HostName 127.0.0.1, Port 2222, UserKnownHostsFile ~/.local/share/dev-vm/known_hosts, StrictHostKeyChecking accept-new`.

**🔵 U3. Внутри гостя не настроен git-identity и swap.**
Агент первым делом коммитит — без `user.name`/`user.email` это падает;
4 GB RAM без swap → OOM на `npm install`/сборке нативных модулей.
*Рекомендация:* пробрасывать `git config user.*` хоста в cloud-config;
добавить zram или swap-файл на 1–2 GB (тонкому диску это почти ничего не
стоит).

**🔵 U4. Дублирование дефолтов README ⇄ скрипт.**
Таблица переменных в README повторяет значения из скрипта — при изменении
одного места второе разойдётся. При текущем размере терпимо; при росте — см.
направление D4.

**🔵 U5. Нет ни CI, ни shellcheck.**
Для bash-проекта shellcheck в GitHub Actions — это 10 строк YAML и ловит
целый класс регрессий бесплатно.

## 4. Долгосрочные направления развития

Отсортированы по отношению «ценность / усилия» для заявленной цели проекта —
дешёвые одноразовые песочницы для агентов.

### D1. Снапшот «pristine» и мгновенный reset (максимальная ценность)

Сейчас «disposable» означает «пересоздавай с нуля» (~5–10 мин с cloud-init).
Правильный жизненный цикл:

```
provision → дождаться cloud-init → снять снапшот "pristine"
reset     → VBoxManage snapshot restore "pristine" (секунды)
```

Это превращает проект из «скрипта установки» в реальный инструмент рабочего
цикла: агент сломал окружение — откатились за 5 секунд. Требует R2
(health-check), чтобы снапшот снимался с гарантированно готовой ВМ.

### D2. CLI с подкомандами вместо одного скрипта

Естественная эволюция: `dev-vm up | ssh | status | snapshot | reset | destroy | ls`.
Сейчас управление размазано по README-сниппетам с `VBoxManage`-заклинаниями.
Один bash-скрипт с `case "$1"` — без смены языка и зависимостей. Сюда же
органично ложатся: авто-подбор SSH-порта (R4), генерация ssh-config (U2),
поддержка нескольких именованных ВМ.

### D3. Вынести cloud-config в шаблон + слой кастомизации

user-data сейчас захардкожен в heredoc (`provision-dev-vm.sh:75-134`).
Вынести в `cloud-init/user-data.tmpl` (подстановка через `envsubst`) и
поддержать drop-in `user-data.local.yaml` — пользователи смогут добавлять свои
пакеты/дотфайлы, не форкая скрипт. Это же решает читаемость: YAML с
экранированными `\$HOME` внутри bash-heredoc — самое хрупкое место файла.

### D4. Верификация артефактов как принцип

Объединить R3/S2/S3 в одну политику: всё, что скачивается, проверяется
(SHA256 образа, подписанный apt-repo NodeSource), версии пиновятся
(конкретный serial образа вместо `current/`, с флагом «взять свежий»).
Это даёт воспроизводимость: сегодняшняя и завтрашняя ВМ идентичны.

### D5. Осознанная история про изоляцию

Задокументировать модель угроз (S5) и, опционально, добавить профили сети:
`NET=full` (как сейчас) | `NET=restricted` (host-only + squid/allowlist на
хосте) для параноидальных сценариев. Не делать сложно заранее — но
зафиксировать в README, что выбор сделан сознательно.

### D6. Стратегический вопрос: своя обвязка vs готовые инструменты

Нишу «поднять одноразовую Ubuntu-ВМ одной командой» закрывают Multipass
(тот же cloud-init, кроссплатформенно, ARM из коробки), Lima, Vagrant.
Честная позиция проекта: **ценность не в поднятии ВМ, а в
специализированном образе для агентной разработки** (CLI-инструменты,
npm-prefix-фикс, дисковая экономия, жизненный цикл reset).
Два следствия:
- не наращивать собственную VM-обвязку сверх необходимого (D2 — потолок);
- рассмотреть Multipass как альтернативный бэкенд: cloud-config уже готов и
  переносим, а проблемы P1/P2/R1 исчезают у бэкенда бесплатно. VirtualBox
  остаётся для тех, кому нужна именно она.

### D7. Гигиена репозитория

shellcheck + shfmt в CI (U5); smoke-тест «скрипт до сборки seed проходит с
фейковым VBoxManage»; CHANGELOG или хотя бы теги. Дёшево, дисциплинирует.

## 5. Бизнес- и продуктовые идеи

Проект сидит в быстрорастущей нише — **безопасное выполнение автономных
агентов**. Облачный сегмент уже занят (E2B, Daytona, Modal, Runloop и т.п.),
но **local-first** угол — «песочницы для агентов на своём железе, без
поминутного биллинга и утечки кода в чужое облако» — заметно свободнее.
Идеи ниже отсортированы от «естественное продолжение» к «отдельный продукт».

### B1. Agent Fleet: локальная ферма агентов (ближайшее продолжение)

Сейчас проект = одна ВМ. Следующий уровень ценности — **оркестрация N
песочниц**: `dev-vm fleet up 5` поднимает пять ВМ из pristine-снапшота (см.
D1), в каждую заезжает агент со своей задачей/веткой, статус — одной командой
или простым TUI/веб-дашбордом (какая ВМ работает, что в логе, сколько диска).
Это превращает утилиту в инструмент реального воркфлоу «раздал агентам задачи
вечером — утром собрал PR'ы». Технически всё уже почти есть: снапшоты,
клоны (`VBoxManage clonevm --options link` — линкованные клоны почти
бесплатны по диску, что бьётся с дисковой философией проекта), авто-подбор
портов (R4).

### B2. Аудит-трейл: «что агент натворил» как фича безопасности

Снапшот перед сессией + diff файловой системы и истории команд после — готовый
ответ на главный страх корпоративного пользователя автономных агентов.
`dev-vm diff` показывает: какие файлы изменены, что ставилось, куда ходила
сеть (через логирование на NAT-стороне хоста). Это дёшево делается поверх D1
и является **уникальной фичей** — облачные песочницы такое почти не дают,
потому что у них нет стимула, а локально это естественно.
Монетизируемый вариант: enterprise-отчёт для комплаенса («агентам можно,
вот контроль»).

### B3. Каталог golden-образов под стеки

Один pristine-образ — хорошо; библиотека — лучше: `dev-vm up --image python-ml`
(CUDA-less PyTorch, uv, jupyter), `--image web` (pnpm, playwright + браузеры),
`--image embedded` (gcc-arm, qemu). Образы версионируются и проверяются по
хэшу (D4). Дистрибуция — просто раздача проверенных cloud-config-профилей
(D3 делает это почти бесплатным) + опционально кэш готовых VDI.
Это и путь к сообществу: профили — то, что люди охотно контрибьютят.

### B4. Локальный бэкенд для фоновых/удалённых агентов

Claude Code и аналоги двигаются к модели «агент работает где-то в фоне».
Ниша: **self-hosted runner** — хост-демон, который принимает задачи
(webhook/CLI/расписание), поднимает ВМ из снапшота, запускает агента
неинтерактивно (`claude -p`), собирает результат в PR и гасит ВМ. По сути
«GitHub Actions runner, но для агентов и на своём железе». Хорошо стыкуется
с B1 и может быть первым по-настоящему продуктовым воплощением проекта.

### B5. Полигон для сравнения агентов (eval-harness)

Уже сейчас образ содержит и Claude Code, и Codex — само напрашивается:
одна задача → две одинаковые ВМ из одного снапшота → два агента → сравнение
результатов (тесты пройдены? diff меньше? время?). Воспроизводимость (D4)
здесь не гигиена, а обязательное условие. Аудитория — команды, выбирающие
агентов, и авторы бенчмарков. Как бизнес слабо монетизируется напрямую, но
даёт видимость и авторитет в нише (см. B7).

### B6. Профили сетевой изоляции как платная/enterprise-фича

Развитие D5 до продукта: `NET=offline` (только localhost), `NET=allowlist`
(прокси на хосте: только api.anthropic.com, github.com, реестры пакетов),
`NET=full`. Плюс B2-логирование. Это ровно то, за что платят организации,
которым «агентам нельзя в интернет, но очень хочется агентов».

### B7. Реалистичная стратегия монетизации

Честная оценка: сам скрипт — не продукт, и конкурировать с облачными
песочницами лоб-в-лоб не стоит. Рабочие пути:

1. **OSS-инструмент → репутация → продукт сверху.** Открытый `dev-vm`
   (brew/apt-пакет, бэкенды VirtualBox/Multipass/Lima), бесплатный навсегда;
   монетизация — хостируемый control plane для B1/B4 или enterprise-фичи
   (B2, B6, SSO, политика).
2. **Нишевой консалтинг/контент.** «Как безопасно запускать автономных
   агентов в компании» — тема, по которой сейчас мало внятной экспертизы;
   проект — рабочее портфолио и генератор лидов.
3. **Не делать:** marketplace образов как отдельный бизнес (не взлетит без
   трафика) и своё облако (капиталоёмко, рынок занят).

Минимальный тест спроса, почти бесплатный: довести пункты 1–6 плана из
раздела 6, оформить README как продуктовый лендинг, выложить на GitHub +
пост в профильные сообщества — и смотреть на issues: чего просят люди,
B1 (ферма), B2 (аудит) или B6 (изоляция).
